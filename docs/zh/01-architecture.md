# 系统架构全景

## 为什么要先谈架构？

在开始写任何代码之前，先理解整体架构，能帮你回答几个关键问题：

- 一个自动化交易系统应该分成几个层次？
- 各层之间如何通信、如何解耦？
- 当我需要加一个新策略时，要动哪些代码、不动哪些代码？

本文将以 **SPX 铁鹰策略（Iron Condor）** 作为贯穿全文的案例，展示一份完整的交易请求在系统中流经的路径。

---

## 一、六层架构

这套系统分为六个逻辑层，从上到下依次为：

```
┌─────────────────────────────────────────────────────┐
│  ① 连接层 (Connection Layer)                        │
│  管理 TWS/IB Gateway 的连接生命周期                  │
├─────────────────────────────────────────────────────┤
│  ② 数据层 (Data Layer)                              │
│  获取市场数据（指数价格、期权链）                     │
├─────────────────────────────────────────────────────┤
│  ③ 策略层 (Strategy Layer)                          │
│  构建交易策略、筛选行权价、计算目标价格               │
├─────────────────────────────────────────────────────┤
│  ④ 执行层 (Execution Layer)                         │
│  下单、跟踪状态、价格退让调整                        │
├─────────────────────────────────────────────────────┤
│  ⑤ 记录层 (Recording Layer)                         │
│  将成交记录持久化到 Excel                            │
├─────────────────────────────────────────────────────┤
│  ⑥ 通知层 (Notification Layer)                      │
│  通过 Webhook 推送成交通知                           │
└─────────────────────────────────────────────────────┘
```

### 各层职责

| 层次 | 核心类（公开版） | 核心职责 |
|:-----|:----------------|:---------|
| 连接层 | `ConnectionManager` | 单例管理连接、固定 ClientID 分配、自动重连 |
| 数据层 | `MarketDataFetcher` + `OptionChainFetcher` | SPX 行情获取、期权链批量请求、OTM 筛选 |
| 策略层 | 用户自定义（本文案例：Iron Condor） | 行权价筛选、信用计算、构建组合合约 |
| 执行层 | `OrderManager` | 组合合约下单、状态跟踪、价格修改 |
| 记录层 | `TradeRecorder` | Excel 自动建表、追加记录、表头维护 |
| 通知层 | `WebhookNotifier` | HMAC 签名、消息推送 |

## 二、以铁鹰策略为例：一次完整的请求流

假设我们执行一个铁鹰策略开仓，SPX 当前价格 5480 点。来看请求如何流经各层：

### 步骤 ①：连接

```
[用户] → 调用 ConnectionManager.connect()
         → 单例检查（已连接则跳过）
         → 分配 ClientID → 连接 TWS:7497
         → 等待 nextValidId → 就绪
```

**为什么用单例？**

在一个自动化系统中，数据获取、期权链查询、订单执行可能由不同模块触发。如果每个模块各自创建一个连接，会引发 TWS 的 ClientID 冲突（"already in use"错误）。单例 + 固定 ClientID 分配保证了所有模块共享同一个连接，且每个模块使用固定 ID，重启后也不会冲突。

### 步骤 ②：获取市场数据

```
[连接层已就绪] → MarketDataFetcher.fetch_snapshot()
               → 创建 SPX 指数合约 (conId=416904)
               → reqMktData(snapshot=true)
               → 回调 tickPrice → 获取 LAST=5480.25
               → 取消订阅 → 断开连接
```

通过 `generate_strike_prices()` 计算潜在行权价范围：

- 下浮 4%：5480 × 0.96 = 5260.8 → 取整到 5265
- 上浮 3%：5480 × 1.03 = 5644.4 → 取整到 5640
- 步长 5 点，共约 76 个行权价

### 步骤 ③：获取期权链数据

```
→ OptionChainFetcher.fetch(expiry="20250919",
                            strikes=[5265, 5270, ..., 5640],
                            underlying=5480.25)
   → 筛选 OTM 合约（Put < 5480.25, Call > 5480.25）
   → 分批 50 个请求/批 → reqMktData(snapshot=true)
   → 等待 bid/ask/delta 数据 → 组装 DataFrame
```

这一步返回的数据样例如下：

| strike | type | delta | bid | ask | mid |
|--------|------|-------|-----|-----|-----|
| 5265 | PUT | -0.03 | 1.20 | 1.35 | 1.275 |
| 5270 | PUT | -0.04 | 1.45 | 1.60 | 1.525 |
| ... | ... | ... | ... | ... | ... |
| 5485 | CALL | 0.03 | 2.10 | 2.30 | 2.200 |

### 步骤 ④：策略构建

策略层拿到期权链数据后，应用铁鹰策略的筛选逻辑：

```
→ 对 PUT 侧：在 OTM put 中找 delta 最接近目标值（如 -0.05）的行权价
→ 对 CALL 侧：在 OTM call 中找 delta 最接近目标值（如 0.05）的行权价
→ 确定 4 条腿：
   - Sell Put @ 目标行权价
   - Buy Put @ 目标行权价 - 5 点（保护腿）
   - Sell Call @ 目标行权价
   - Buy Call @ 目标行权价 + 5 点（保护腿）
→ 计算理论信用 = (卖Put中间价 - 买Put中间价) + (卖Call中间价 - 买Call中间价)
→ 检查信用是否在目标区间
→ 输出策略字典：{put_sell_strike, put_buy_strike, call_sell_strike, call_buy_strike, ...}
```

注意：**这一段逻辑的具体参数（Delta 阈值、信用区间等）是本系统的核心策略参数，不在公开范围内**。公开的 `option_chain.py` 提供了 `find_strikes_by_delta()` 这个工具方法，但具体的目标值由用户在策略层自行决定。

### 步骤 ⑤：执行订单

```
[策略已构建] → OrderManager 连接 TWS
             → 创建 BAG 组合合约（4 条 ComboLeg）
             → 创建 Order(action=BUY, orderType=LMT, price=计算值)
             → placeOrder() → 注册到 active_orders
             → 进入监控循环：
                每 1 秒检查 orderStatus
                每 N 分钟检查是否需要价格退让
                成交 → 记录成交价
                超时 → 返回失败
```

### 步骤 ⑥：记录成交

```
[订单成交] → TradeRecorder.record({
               date, time, expiry,
               put legs, call legs,
               fill_price, quantity
           })
           → 自动创建/打开 Excel → 追加行 → 格式化日期 → 保存
```

### 步骤 ⑦：发送通知

```
[记录完成] → WebhookNotifier.send_trade_update(
               title="Iron Condor Filled",
               execution_time="2025-09-03 15:45:00 ET",
               details="... 市场数据 ..."
           )
           → 生成 HMAC-SHA256 签名 → POST → 钉钉机器人推送
```

### 全流程时序图

```
用户    Connection   MarketData   OptionChain   策略层    OrderManager   Recorder  Notifier
 │          │            │            │           │          │             │          │
 ├─connect()→│            │            │           │          │             │          │
 │          ├─snapshot()→│            │           │          │             │          │

```

## 三、涉及的设计模式

这套系统采用了 4 种经典设计模式，每种模式的选用都有明确的实战原因：

| 模式 | 使用场景 | 为什么需要 |
|:-----|:---------|:----------|
| **单例** | `ConnectionManager` | 保证全局只有一个 TWS 连接，避免 ClientID 冲突 |
| **工厂** | 策略选择（未公开） | 运行时根据用户选择创建不同的策略实例 |
| **策略** | 策略基类 + 子类 | 每种交易策略（铁鹰、垂直价差等）可独立实现和替换 |
| **模板方法** | `BaseStrategy.build_strategy()` | 定义策略构建的标准流程骨架，子类填充细节 |

## 四、架构要点总结

1. **分层解耦**：各层通过明确的接口通信，数据层不知道策略层在算什么，执行层不关心策略逻辑
2. **单例连接**：所有模块共享一个 TWS 连接，避免多连接冲突
3. **策略可插拔**：新增策略只需实现策略基类接口，无需改动其他层
4. **错误隔离**：记录失败不中断交易、通知失败不中断记录——每层独立容错
5. **配置驱动**：端口、超时、重试次数等全部参数化，不硬编码
